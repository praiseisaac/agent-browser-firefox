agent-browser-firefox — Architecture & Flow

A Firefox-native browser automation CLI for AI agents, driving Firefox over WebDriver BiDi — the counterpart to agent-browser's Chrome/CDP attachment.

1 System architecture

Each CLI command is a short-lived process. The browser session lives in a long-running daemon, so state (the open tab, @refs) survives across invocations. The daemon talks to Firefox over a BiDi WebSocket.

Firefox / daemon boundary WebDriver BiDi WebSocket Local IPC (Unix socket)

2 Command flow & the “no-restart” rule

Every command calls ensure_daemon. While a daemon is alive, all commands reuse the same Firefox — it is never restarted. A restart only happens after a daemon dies, because Firefox orphans its single BiDi session.

3 BiDi handshake (cold start)

Discovery is deterministic: we choose the port, so the endpoint is ws://127.0.0.1:<port>/session. We poll the TCP port rather than parse stderr — on macOS the Firefox launcher closes its inherited stderr after startup.

4 Instance lifecycle

Instances are addressable by ID (the --session name) and persisted to a registry on disk, so they can be listed, resumed, and closed across separate terminals.

5 Configuration precedence

Easy setup: every setting has a sane default and can be overridden at four layers. Later layers win.
